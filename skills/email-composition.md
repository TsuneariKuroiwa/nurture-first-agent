# Skill: Email Composition (メール作成)

## Purpose
ユーザーに代わってビジネスメール・研究関連メールを作成する。カレンダー・Slack・Gmailから文脈を収集し、Gmailの下書きとして作成する。送信はユーザー自身が行う。

## Escalation Criteria

### 自律実行OK
- カレンダー・Slack・Gmailからの情報収集
- ユーザーの指示に基づく下書き作成
- 送信済みメールと下書きの差分確認（学習目的）

### ユーザー確認が必要
- 宛先・CC の判断が曖昧な場合
- メール本文に含める具体的内容が不明確な場合
- 謝罪・クレーム対応など繊細な内容

### 即エスカレーション
- 外部組織への初回コンタクト（ユーザーの名前で送る以上、慎重に）
- 金銭・契約に関わる内容
- 機密情報を含む可能性がある場合

## Workflow

### 1. 文脈収集
メール作成の依頼を受けたら、以下のソースから情報を集める:
- **Google Calendar**: 関連するイベント（日時・場所・参加者・説明）
- **Slack**: 関連チャンネル・DMでの直近のやり取り
- **Gmail**: 過去のメールスレッド、宛先のメールアドレス
- ユーザーへの質問は最小限にする。まず自分で調べて、足りない情報だけ聞く

### 2. 確認事項の整理
情報収集後、以下をユーザーに確認する（不明な場合のみ）:
- **宛先**: To / CC の振り分け
- **目的**: メールの主旨（お礼、依頼、報告、相談など）
- **盛り込む内容**: 具体的に触れるべきポイント
- **トーン**: 相手との関係性に応じた敬語レベル

### 3. 下書き作成
- Gmail下書きとして作成する（`gmail_create_draft`）
- 署名は利用者が別途設定する fixed info を使用する
- ユーザーが確認・編集してから送信する前提

### 4. 送信後の差分確認
ユーザーが送信した後、依頼があれば送信済みメールを読んで下書きとの差分を確認する。修正箇所は今後の学習に活かす。

## Writing Style Guidelines

### 基本方針
- **簡潔**: 不要な修飾・フィラー表現は入れない
- **具体的**: 曖昧な表現より正確な用語を使う
- **柔らかい断定**: 共有の結論には「〜かと思います」「〜と認識しております」など
- **構造**: 挨拶 → 本題 → 依頼/次のアクション → 締め → 署名

### 避けるべきパターン
- 不必要な時系列マーカー（「後半に」「最後に」など、文脈で明らかな場合）
- 過剰な対応表明（「メールでもオンラインでも対応いたします」のような蛇足）
- 一文に情報を詰め込みすぎる（箇条書きや文分割を検討）

### Slack DM の書き方
- 「お疲れ様です」で始める（日本語ビジネス文脈）
- 敬語を使うが過剰に丁寧にしない（「ご教示いただけますと幸いです」は重すぎ）
- 太字・見出し・カテゴリラベルは使わず、フラットな箇条書き
- 物品・ツールは用途で書く（商品名・価格はリンクに任せる）
- 相手側が決定権を持つ話題では、自分側の予算・判断リスクを先回りして書かない

### 名乗り（自己紹介）の使い分け

文脈に応じて所属を変える。例:
- 社外・初回コンタクト → 所属組織名をフルで書く
- 既に関係のある相手 → 名前のみ or ごく簡潔な所属
- 同一組織内 → 名前のみ

具体的なテンプレートは利用者が自分の Contact List / Fixed Information 側で管理する。

### 敬語レベルの目安
| 相手 | レベル | 例 |
|------|--------|-----|
| 外部組織・企業の方 | 丁寧 | 〜いただければ幸いです、〜のほどよろしくお願いいたします |
| 社外の上位者 | 丁寧 | 同上 |
| 組織内の上司・指導教員 | やや丁寧 | 〜でしょうか、〜お願いします |
| 事務局（日常的） | やや丁寧・親しみ | 簡潔な自己紹介、宛名「担当者様」、署名「姓のみ」 |
| 同僚・チームメンバー | カジュアル丁寧 | 〜ですね、〜お願いします |

## Fixed Information（利用者ごとに用意）

このスキルは雛形であり、以下の「Fixed Information」と「Contact List」は
利用者が自分のローカル環境でメンテナンスするべきもの。

### 署名テンプレート（例）

```
===============================================
<組織名>
<部署・役職>
<氏名>
Email: <primary>, <secondary>
===============================================
```

利用シーン:
- 外部組織への初回コンタクト、フォーマルメール、CC/宛先が多いメール → フル署名
- 日常的にやり取りする相手・1:1 メール → 姓のみ or 省略

### Contact List（例フォーマット）

```markdown
### <グループ名>
| 名前 | 所属 | メールアドレス | 備考 |
|------|------|---------------|------|
| ... | ... | ... | ... |
```

※ 実際の連絡先は公開リポにコミットせず、`profile.md` 相当のローカルファイルで管理すること。

## Learned Patterns

### LP-EMAIL-001: 送信後の差分確認で文体を学習する
- **適用条件**: ユーザーがメールを送信した後に差分確認を依頼された場合
- **行動指針**: 下書きと送信済みメールを比較し、修正箇所を具体的にリストアップする。修正の傾向を把握して今後の下書きに反映する

### LP-EMAIL-002: 断定を避けて柔らかい表現を使う
- **適用条件**: ミーティングの議論内容や合意事項をまとめる場合
- **行動指針**: 「〜ました」ではなく「〜かと思います」「〜と認識しております」を使う。特に相手と共有する認識については断定を避ける

### LP-EMAIL-003: 蛇足を削ぎ落とす
- **適用条件**: メール本文の仕上げ時
- **行動指針**: 「メールでもオンラインでも対応いたします」のような、なくても意味が通る表現は削除する。相手が読む時間を尊重する

### LP-EMAIL-004: 用途ベースの説明を優先、商品名・型番は本文に書かない
- **適用条件**: メール・Slack DM で物品・ツール・機器を挙げるとき
- **行動指針**: 商品名やモデル番号ではなく用途（「オンライン会議用のワイヤレスイヤホン」「フィールド用の靴」）で説明する。商品名・型番・価格はリンクに任せ、本文には書かない。読み手は「何のために必要か」が分かれば判断できる。

### LP-EMAIL-005: Slack DM は過剰構造化しない（フラットな箇条書き）
- **適用条件**: Slack DM（特に上位者への相談系）を書くとき
- **行動指針**: 太字セクション見出し、カテゴリラベル、マークダウンの `##` 見出しを使わない。フラットな箇条書きで十分。DM はメールより会話的で、構造化されたドキュメント風レイアウトは重い印象を与える。

### LP-EMAIL-006: 関係性が近い相手には自己紹介と用務を具体化して簡潔に
- **適用条件**: 日常的にやり取りする相手（事務局、同僚等）にメールを書くとき
- **行動指針**: フル所属は不要。関係性を示す簡潔な自己紹介にする。用務内容は一般語より具体的な名詞を使う。署名も姓のみ、宛名は「○○様」のみ。

## Error Cases

### EC-EMAIL-001: 購入主体が相手側の話題に、自己側の規定を混在させた
- **事象**: 上位者が組織予算で購入すると表明した話題で、別文脈（自分が立替購入する場合の規定）を機械的に併記してしまい、文脈矛盾を指摘された。
- **原因**: 参照資料から関連情報を拾うとき、「この情報が今の主体に適用されるか」を検証せず、並列提示した。
- **回避策**: 情報を複数提示するとき、それぞれの情報が「誰の行動に関わるか」をタグ付けして確認する。規定を網羅的に出す前に「今の話題に関係するか」を 1 段階フィルタする。
