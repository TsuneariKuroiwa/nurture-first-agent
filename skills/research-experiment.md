# Skill: Research & Experiment

## Purpose
ユーザーの研究スタイルに基づいて実験の設計・実行・分析を支援する。

## Escalation Criteria

### 自律実行OK
- 実験コードのボイラープレート生成
- データの前処理・可視化
- 既知のフレームワークでの実験実行
- 結果の集計・テーブル作成

### ユーザー確認が必要
- 実験設計の決定（何を比較するか、ベースラインは何か）
- ハイパーパラメータの選択方針
- 結果の解釈・結論の導出
- 論文への記載内容

## Research Domain

ユーザーの研究領域に応じて書き換えてください（例）:
- 機械学習・深層学習
- 時系列データ解析、分類問題
- Foundation model / 事前学習

## Experiment Design Patterns

- クラス/ラベル分割設計では、物理的解釈の明確さとシンプルさを優先。最終判断はユーザーに仰ぐ。
- しきい値や境界値はドメイン知識に基づく判断であり、必ずユーザーに確認する。
- 検討過程の図も最終決定版と別に保持する（再検討時に役立つ）。
- レポートでは内部ID（label_id等）ではなく人間可読なラベル名で表現する。

## Tool Preferences
- 言語: Python
- MLフレームワーク: PyTorch
- 可視化: matplotlib / UMAP / t-SNE
- 実験管理: スクリプトベース（.sh + YAML config）
- GPU並列: bash `&` で複数GPU並列実行、`wait` で完了待ち

## Learned Patterns

### LP-RE-01: 技術提案前に根本原因を確認
- **適用条件**: 代替手法・センサー・アルゴリズムを提案するとき
- **行動指針**: (1) 現状の失敗原因を明確にする、(2) 提案がその原因を本当に解決するか論理的に検証する、(3) 同じ検出原理の代替はボトルネックをバイパスできない場合がある。
- **根拠**: 2026-03-22 15:30 (ガス濃度方式の限界 vs PM2.5の粒子カウント方式)

### LP-RE-02: データ分析の提案前にデータスキーマを確認
- **適用条件**: データ分析・実験条件の提案をするとき
- **行動指針**: 分析に必要なデータが実際に記録されているか、CSVカラムや記録フォーマットを確認してから提案する。
- **根拠**: 2026-03-22 20:00 (換気扇状態がCSVに未記録のまま分析提案)

### LP-RE-03: 技術課題は最大ブロッカーを最初に提示
- **適用条件**: 技術的な実現可能性を議論するとき
- **行動指針**: 課題を並列に列挙せず、最大のブロッカー（ライブラリ/フレームワークの制約等）を最初に特定・提示する。「これが一番の壁で、これをクリアすれば残りは対応可能」という構造で話す。
- **根拠**: 2026-03-22 20:10 (Aggregatorパターン対応がボトルネック)

### LP-RE-04: キャッシュは計算コストが高い部分だけ
- **適用条件**: スクリプトにキャッシュ機構を追加するとき
- **行動指針**: ラベル等の軽い派生データはキャッシュに含めない。キャッシュに含めると、ラベル変更のたびに`--no-cache`が必要になる悪設計になる。
- **根拠**: 2026-03-22 23:00 (features+embeddingsのみキャッシュ、ラベルは毎回生成)

### LP-RE-05: 質疑応答の回答構成
- **適用条件**: 学会発表・学位論文の質疑応答準備
- **行動指針**: 「質問者の指摘を受け止める → あえてその選択をした理由をポジティブに述べる」の構成。技術用語は口頭で伝わるか意識する（閉形式→解析的に計算できる）。箇条書きではなく自然な文章の流れにする。
- **根拠**: 2026-03-22 23:30 (正規分布採用の質疑応答)

### LP-RE-06: 企業名・品番は公式ソースで検証
- **適用条件**: 論文・レポートに企業名・品番を記載するとき
- **行動指針**: 元情報（メール等）も誤りうるため、必ず公式サイトで正確な表記を確認する。
- **根拠**: 元情報のスペルミスに引きずられて論文に誤った企業名を書きそうになった事例。

### LP-RE-07: kNN局所純度によるクラス分離度の定量評価
- **適用条件**: 特徴空間上のクラス分離度を定量評価するとき
- **行動指針**: 重心距離ではなくkNN（k=50程度）ベースの混同行列を計算する。各クラスの局所純度（近傍に占める自クラス割合）とクラス間混在率を算出。複雑形状のクラスタには重心距離は不適で、kNNは形状に依存しないためより信頼性が高い。Silhouette scoreより「どのクラスと混ざっているか」が直接わかる点で有用。
- **根拠**: 2026-04-10 15:00 (Swimming 3分割の評価、ユーザーが重心距離を不適と指摘)

### LP-RE-08: ラベルマージは特徴空間分離度＋生物学的重要性の両面で判断
- **適用条件**: 特徴空間上で区別困難なクラスのマージを検討するとき
- **行動指針**: 局所純度が低くても、ドメイン上の重要性が高いクラスは独立クラスとして維持する。外部的な識別可能性（映像・追加センサー等）も考慮。ドメイン的に関心が低いクラスは Other 等へマージして問題ない。最終判断はユーザーに確認。
- **根拠**: ドメイン上重要なクラスを機械学習的な分離度だけで判断してマージしてしまうと、後工程の解釈に支障が出る。

### LP-RE-09: ラベル定義プロセス
- **適用条件**: 新しい行動ラベルを定義・変更するとき
- **行動指針**:
  1. 先行研究で確立された用語を調査する
  2. 造語の場合は「独自定義」と明示する
  3. 名称・定義文をユーザーに確認してから記述する（見つからない≠推測して書いてよい、ではない）
  4. 定義文は簡潔に。自明な否定文（例:「真の潜水は伴わない」）は不要
  5. 先行研究との対応関係（例: 用語A ≈ 既知用語B）も記録する
- **根拠**: 新ラベルを定義する際、先行研究の確立用語と独自造語を混同した事例。

### LP-RE-10: ノートブック用ライブラリをスクリプトから呼ぶときは plt.show() を無効化
- **適用条件**: ノートブック前提で書かれた可視化ユーティリティ（内部的に `plt.show` を呼ぶタイプ）をバッチスクリプトから呼ぶとき
- **行動指針**: スクリプト冒頭で `plt.show = lambda *a, **k: None` とモンキーパッチして `plt.show()` を無効化する。ライブラリが内部で `plt.savefig` + `plt.show` するパターンでは、`plt.show` が figure を閉じてしまい後続処理に影響する。ライブラリ自身が保存先を持っている場合は、そこから成果物 PNG をコピーするのが最も確実。`plt.close('all')` で消してしまわない。

### LP-RE-11: サイレント失敗を防ぐため、ラベル変換後にアサーションを入れる
- **適用条件**: ラベル/種名/ID 等の変換マップを通した後にマッチングや集計を行うとき
- **行動指針**: 変換後のラベルが期待する語彙に含まれているかアサーションする。例えば `utils.convert_numpy_labels_str` が大文字始まり前提なのに小文字を渡すと、マップ失敗で数値 ID が文字列として残り、その後のマッチングが全部 0 になるがエラーは出ない。`assert all(l in EXPECTED_LABELS for l in converted)` のような境界チェックでサイレント失敗を早期検出する。
- **根拠**: 大文字/小文字の不一致（例: `"foo"` vs `"Foo"`）で変換マップが素通りし、集計結果が全て空になる事故があった。

### LP-RE-12: 比較用（mutual 版）ラベルマッピングは旧版との整合性を最優先
- **適用条件**: ラベル体系を更新し、新旧比較のための mutual 版マッピングを作るとき
- **行動指針**: 新ラベルのドメイン上の意味より、旧ラベルが属していたカテゴリとの一貫性を優先する。ラベルの性質だけ見れば別カテゴリにしたくなる場合でも、統計的比較可能性のほうが重要。mutual 版の目的は「過去の結果と同じ枠組みで比較できること」である。
- **根拠**: ユーザーからの指摘で分類を修正した事例。

## Error Cases

### EC-RE-01: GPU compute capability不一致のサイレントフェイル
- **事象**: PyTorch 1.13.1+cu117がsm_120 GPUに対応しておらず、エラーなしに誤った結果を返した。
- **回避策**: 新しいGPU環境では必ずCPU出力と照合して検証。PyTorchバージョンとGPU compute capabilityの整合を確認。

### EC-RE-02: パッケージforce-reinstallの連鎖的副作用
- **事象**: PyTorch force-reinstall → numpy 2.x → pandas非互換 → numpy downgrade → sklearn downgradeでAPIが壊れた。
- **回避策**: force-reinstall後は主要ライブラリのバージョンを全チェック。requirements.txtでバージョン固定を検討。

### EC-RE-03: コンテナ環境のパス不一致
- **事象**: 古い環境で生成されたYAML configのパスが現環境と異なり、データが0件でロードされた。
- **回避策**: コンテナ環境のパスマッピングは最初に確認。一括置換は件数を提示してユーザーの承認を得てから実行。

### EC-RE-04: プロジェクト固有の定義を確認なしで推測記述
- **事象**: あるラベルの定義をプロジェクト内で見つけられず、先行研究から推測で比較表に記載。ユーザーに「そもそも定義知ってた？」と問われ、推測で書いたことを指摘された。
- **原因**: 「見つからない→自分で推測して書いてよい」と誤った判断。
- **回避策**: LP-RE-09に従う。プロジェクト固有の定義は見つからない場合、ユーザーに確認してから書く。推測で書かない。
