# Skill: Harness Engineering（自律開発ハーネス）

## Purpose
エージェント（Claude Code等）に自律的にコーディングタスクを遂行させるための
「ハーネス（環境）」を設計する。「どうやって作るか」を指示するのではなく、
「間違った作り方をしたら即座に検出できる仕組み」を整備することで、エージェントの
自律性を高める。

## 出典
Qiita記事「ハーネスエンジニアリング」（ユーザー参照）+ ユーザーによる3エージェント拡張

## 核心思想: 制約設計 > 手順指示

エージェントの自律性を高めるには、指示を増やすのではなく制約（フィードバックループ）
を強化する。

CLAUDE.mdの規約 + 型チェック + テスト + lint = 制約の層。
エージェントはこの制約の中で自由に実装方法を選ぶ。

ユーザーが制御するのは「制約の設計」であって「手順の指示」ではない。
カバレッジゲートも制約の一つ。

## Escalation Criteria

### 自律実行OK
- 既存ハーネスの軽微な調整（feature_list.jsonへの項目追加等）
- エージェントファイルのルール追記
- フィードバックループの拡充（lintルール追加等）

### ユーザー確認が必要
- ハーネスの新規構築
- エージェント構成の変更（2→3エージェント等）
- ブランチ戦略の変更
- プロジェクト全体のCLAUDE.md規約変更

## ハーネスの5構成要素

1. **環境初期化スクリプト** (`init.sh`)
   - 依存インストール、DB初期化、フィクスチャ投入を1コマンドで完了
   - エージェントがクリーン状態から作業を開始できる
2. **インクリメンタル進捗管理** (`feature_list.json` + `claude-progress.txt`)
   - 機能を細かく分解、進捗を機械可読な形式で管理
   - 人間の介入なしにエージェントが進捗を追跡・更新できる
3. **4層フィードバックループ**
   - 型チェック / lint / 単体テスト / E2E or 手動検証
   - 各層で間違いを即座に検出する
4. **コードベース即コンテキスト** (`CLAUDE.md`)
   - プロジェクト規約、API仕様、既知の落とし穴を集約
   - エージェントが必要な文脈をプロジェクト内から取得できる
5. **技術的負債制御**
   - カバレッジゲート、設計原則の自動チェック
   - 負債の蓄積を防ぐ

## エージェント構成

### 2エージェント案（記事原案）
- イニシャライザー / コーディング

### 3エージェント案（カスタム拡張・推奨）
- **実装エージェント** (`.claude/agents/implement.md`): コードを書く
- **評価エージェント** (`.claude/agents/evaluate.md`): ブラウザ動作確認・テスト実行・PRレビュー
- **オーケストレーター** (`.claude/agents/orchestrator.md`): 複数機能の進捗管理、ブランチ戦略の実行

**3エージェント案を採用する理由**:
ブラウザ評価（Claude in Chrome）はコンテキストを大量消費する。実装と評価を同一
セッションで回すとコンテキスト爆発が起きる。分離することで各セッションのコンテキストを
軽く保てる。モデルは`opus`を推奨（`sonnet`より判断の質が高い）。

## ブランチ戦略

### 依存がある機能
- **ブランチ積み上げ方式**: 前のブランチから次のブランチを切る
- PRは機能ごとに作成（差分が小さくレビューしやすい）
- マージは最後のブランチ→mainで一括

### 独立した機能
- mainから直接ブランチを切る
- 並列実行可能

### 設計原則
ハーネスの運用設計は、ユーザーのレビュースタイル（夜間一括実行→翌朝まとめてレビュー等）
から逆算して決める。レビュースタイルが先、ブランチ戦略は後。

## 技術的制約と回避策

### サブエージェントとMCPの非互換
カスタムサブエージェント（`.claude/agents/`）からは Claude in Chrome（MCP）に
アクセスできない。GitHub Issue #13898 を参照。

**対策**: 評価エージェントでブラウザ検証が必要な場合、MCPアクセス不可なら
`passes:false`のまま残し、トップレベルセッションで実行する。APIテスト・コードレビュー
だけで`passes:true`にしない—ハーネスの品質保証が骨抜きになる。

### Windows + Git Bash環境のパス問題
`cd backend;C:\...` のようなWindowsパスは、bashで`;`がコマンド区切りとして
解釈され、`backend;C`のようなゴミディレクトリを生成する。

**対策**: エージェントファイルに「Windowsパス禁止、Unixパス・相対パス優先」を明記する。

## Learned Patterns

### LP-HE-01: 制約設計を手順指示より優先
- **適用条件**: エージェントに自律タスクを任せるとき
- **行動指針**: 「どう作るか」を細かく指示せず、「間違ったら検出される仕組み」を整える。型チェック・テスト・lint・カバレッジゲートで制約の層を作る。エージェントはその制約の中で実装方法を自由に選ぶ。
- **根拠**: 2026-04-08 11:00

### LP-HE-02: 実装と評価のエージェント分離
- **適用条件**: ブラウザ検証を含む自律開発タスク
- **行動指針**: 実装エージェントと評価エージェントを別セッションに分離する。ブラウザ評価はコンテキストを大量消費するため、同一セッションで回すとコンテキスト爆発が起きる。さらに複数機能の並行開発にはオーケストレーターを追加する。
- **根拠**: 2026-04-08 10:00

### LP-HE-03: ブランチ戦略はレビュースタイルから逆算
- **適用条件**: 複数機能の自律開発でブランチ戦略を決めるとき
- **行動指針**: 「夜間一括実行→翌朝まとめてレビュー」スタイルなら、依存機能はブランチ積み上げ＋PRは機能ごと＋マージは一括。レビュースタイルが先で、ブランチ戦略はそれに合わせる。
- **根拠**: 2026-04-08 12:00

### LP-HE-04: feature_list.jsonには「何を作るか」のみ書く
- **適用条件**: ハーネス用の機能リスト設計
- **行動指針**: stepsには「何を作るか」は書くが「どう作るか」は書かない。実装方法はエージェントが制約の中で選ぶ。指示の詳細度を上げても自律性は上がらない—フィードバックループの強化で制御する。
- **根拠**: 2026-04-08 10:00/11:00

## Error Cases

### EC-HE-01: サブエージェントからMCPアクセス不可で評価が骨抜きに
- **事象**: 評価エージェント（`.claude/agents/evaluate.md`）をサブエージェントとして起動した場合、Claude in Chrome（MCP）にアクセスできない。GitHub Issue #13898。
- **回避策**: ブラウザ検証が必要な場合は`passes:false`で残し、トップレベルセッションで実行する。APIテスト・コードレビューだけで`passes:true`にしない。評価エージェントに「MCP不可時は不合格として記録」ルールを明記する。

### EC-HE-02: Windows + Git Bash でゴミディレクトリ生成
- **事象**: サブエージェントがBashコマンドでWindowsパス（`C:\...`）を使用した際、`;`がシェルのコマンド区切りとして誤解釈され、`backend;C`/`frontend;C` のようなゴミディレクトリが生成された（再現性あり）。`cd backend;C:\...` → `cd backend` + `C:\...`（後者が空ディレクトリ作成の原因）。
- **回避策**: 3つのエージェントファイル全てに「Windowsパス禁止、Unixパス・相対パス優先」ルールを明記する。Windows + Git Bash環境ではパスの扱いに要注意。
