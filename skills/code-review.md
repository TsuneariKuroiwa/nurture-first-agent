# Skill: Code Review

## Purpose
ユーザーの基準に基づいてコードレビューを実施する。

## Escalation Criteria

### 自律実行OK
- Learned Patterns に合致する指摘（既知のパターン）
- 静的解析で検出できる明確な問題（型エラー、未使用変数など）
- フォーマット・命名規則の指摘

### ユーザー確認が必要
- アーキテクチャ上の判断を伴う指摘
- 「マージ可否」の最終判断
- パフォーマンスとのトレードオフがある指摘
- Learned Patterns に前例がないケース

## Rock's Review Style

- レビューコメントはインラインが基本。全体に共通する一般的な内容のみ通常コメント（pr comment）で投稿する。
- コード特定箇所への指摘 → インラインコメント（reviews API + comments配列）
- 全体共通の案内 → 通常コメント（pr comment）
- 抽象的なコメントは避け、変更前後のコード例をセットで提示する。
- プロジェクト横断の問題は個別PRで指摘せず、issueに切り出す。PRコメントでは「別issueで対応予定」と記載。
- **投稿前承認必須**: レビューコメントはGitHubに投稿する前に必ずユーザーの承認を取る。投稿済みコメントはAPI経由で削除できないため、不要コメントがあるとユーザーが手動削除する手間が発生する。
- **下書き提示形式**: ユーザーに確認を取る際は (1) 該当コードの引用 (2) 投稿される文面そのまま を含めて提示する。ユーザーが修正指示を出しやすい形にする。
- **トーン**: ユーザーはPM・発案者。開発メンバーはユーザーのアイデアに共感して自発的に参加している後輩たち。レビューは簡潔・建設的に。「お疲れ様でした」等のねぎらい定型句は不要。上からの評価ではなく協力者としての敬意を持つ。

## Common Feedback Patterns

- 「影響範囲を正確に記述すること」— 「上書き」「無効化」等の強い表現を使うときはスコープ（グローバル/ローカル）を必ず明記。
- 「その判断は私に言われなくてもできるようになると良い」— 新しいコードが追加されたとき、既存コードとの一貫性を自主的にチェックすべき。
- 「表現に問題があった場合、修正すべきは表現であって項目の削除ではない」— 指摘の正確性を問われても萎縮せず、表現を直して妥当な指摘は残す。

## Learned Patterns

### LP-CR-01: 既存コードとの一貫性チェック
- **適用条件**: PRで新しいコードが追加されたとき
- **行動指針**: 同じファイル内の既存パターン（設定取得方法、import形式、命名規則など）をgrepで確認し、不一致があれば指摘する。
- **根拠**: 2026-03-22 18:30 (PR#100 os.environ.get vs env()), 18:45 (PR#122 permission_classes), 14:00 (PR#143)

### LP-CR-02: diffだけでなく最新コードを確認
- **適用条件**: PRの修正対応確認、過去コメントの対応状況確認
- **行動指針**: `git show origin/<branch>:<file>` で最新ファイルを確認してから判定する。diffの追加行だけを見て判断しない。
- **根拠**: 2026-03-22 18:45 (未対応と誤判定), 18:00 (修正確認手順)

### LP-CR-03: プロジェクト横断の問題はissueに切り出す
- **適用条件**: PRレビューで発見した問題がプロジェクト全体に共通するとき（マジックナンバー、設定パターンの不統一等）
- **行動指針**: 個別PRで指摘せず、GitHub issueに切り出す。PRコメントでは「別issueで対応予定」と記載。
- **根拠**: 2026-03-22 14:00 (role定数 → issue#146), 15:00 (permission_classes → issue#147), 18:00 (パターン定着確認)

### LP-CR-04: レビューコメントは具体例付き
- **適用条件**: レビューコメントを書くとき（常に）
- **行動指針**: 変更前後のコード例をセットで提示する。抽象的な説明だけでは「どういうこと？」と聞かれる。
- **根拠**: 2026-03-22 14:00 (「上に配置」が抽象的で伝わらなかった), 15:00 (DRF内部動作の段階的説明)

### LP-CR-06: 仕様レベルの設計漏れを指摘する
- **適用条件**: バグやエッジケースを発見したとき
- **行動指針**: 表面的な症状（ロジックの修正）だけでなく、仕様レベルの設計漏れ（パラメータ不足、APIインターフェースの欠陥等）がないか確認する。「そもそもこのパラメータは必要ではないか」という視点を持つ。
- **根拠**: 2026-03-27 14:00 (月指定のエッジケース → yearパラメータが仕様に存在しない根本原因)

### LP-CR-07: 推測よりテスト実行で確認
- **適用条件**: バグの有無や動作をレビューで判定するとき、テスト実行可能な環境があるとき
- **行動指針**: Docker等のテスト環境が利用可能なら、推測でコメントするより実際にテストを実行して確認する。「テスト実行し確認済み」と記載できれば説得力が増す。
- **根拠**: 2026-03-27 16:00 (Docker環境でAttributeErrorを再現確認)

### LP-CR-08: 並行エージェントで複数PR効率化
- **適用条件**: 複数PRのレビュー・対応状況チェックを行うとき
- **行動指針**: 各PRのdiff取得・コメント確認を並行エージェントで実行し、結果をまとめて提示する。
- **根拠**: 2026-03-27 10:00 (7PR一括チェック、5エージェント並行)

### LP-CR-05: gh CLIのbody渡しは--body-file
- **適用条件**: gh issue create / gh pr review 等でMarkdownコードブロック・バッククォート・日本語を含むbodyを渡すとき
- **行動指針**: インラインではなく `--body-file` で一時ファイルから読み込む。review_body.jsonパターンを統一的に使う。
- **根拠**: 2026-03-22 14:30 (バッククォートがシェルに食われた)

### LP-CR-09: 指摘前にフレームワークのデフォルト動作を検証する
- **適用条件**: エラーハンドリング、ロギング、バリデーション等の追加を指摘しようとするとき
- **行動指針**: フレームワーク（Django, DRF等）のデフォルト動作で十分でないかを自律的に検証してから指摘する。「あった方がベター」レベルの指摘はノイズ。
- **根拠**: 2026-03-30 14:00 (transaction.atomicのtry/except → DRFデフォルトハンドラで十分)

### LP-CR-10: 再指摘は既存コメントへの返信で簡潔に
- **適用条件**: レビューで指摘した項目の修正が不完全・未対応だったとき
- **行動指針**: 新しいレビューコメントではなく、既存のインラインコメントに返信する形で簡潔に依頼する。
- **根拠**: 2026-03-29 14:00 (PR#145エラーメッセージ指摘の再指摘)

### LP-CR-11: 各指摘に推奨アクションレベルを付ける
- **適用条件**: コードレビューの指摘リストをユーザーに提示するとき
- **行動指針**: 各項目に「安全に即修正可能」「影響調査後に修正」「コメント追加のみ」等の推奨アクションを付ける。ユーザーの判断パターン: 可逆的→即OK、動作変更→調査要求、不明→質問、低リスク→軽い対応。
- **根拠**: 2026-04-01 14:00, 14:10 (サンプルプロジェクト レビューでのユーザーの応答パターン)

### LP-CR-12: 複数PRのマージ戦略
- **適用条件**: 複数のPRを連続でマージするとき
- **行動指針**: mergeStateStatus=CLEANなPRから先にマージ。コンフリクト解消では各PRが担当する「機能領域」を基準に採用元を判断する。
- **根拠**: 2026-03-30 15:00, 15:30 (PR#100→#110→#101マージ)

## Error Cases

### EC-CR-01: 影響範囲の誤記述
- **事象**: `permission_classes = []` を「全てのパーミッションを無効化する」と記述し、グローバル設定の変更と誤解される表現になった。実際はそのビュークラスだけの設定。
- **回避策**: 「上書き」「無効化」等の表現ではスコープを必ず明記（「このビュークラスにおいて」等）。

### EC-CR-02: 過剰な萎縮による妥当な指摘の削除
- **事象**: 指摘の表現が不正確だと指摘された後、妥当な指摘ごと項目を削除してしまった。
- **回避策**: 表現の修正 ≠ 項目の削除。指摘自体が妥当なら表現を直して残す。

### EC-CR-04: ユーザー承認なしのレビュー投稿
- **事象**: レビュー内容に自信があったため、ユーザーの確認なしにGitHub APIで投稿。不要なコメントが含まれており、ユーザーが手動で削除する手間が発生。
- **回避策**: Phase 1では全ての外部アクション（GitHub投稿、メール送信等）について事前にユーザーの承認を取る。投稿済みコメントはAPI経由で自動削除できない。

### EC-CR-05: typo指摘に不要な理由を付けて説得力低下
- **事象**: テスト関数名typoの指摘で「grepで引っかからなくなるので」と弱い理由を追加。
- **回避策**: typoはtypoとして簡潔に指摘する。弱い理由は全体の説得力を下げる。

### EC-CR-03: 既存コード一貫性の見落とし
- **事象**: settings.pyで `os.environ.get` が唯一の例外だったが、既存が全て `env()` であることに気づかなかった。ユーザーに指摘されてから確認。
- **回避策**: LP-CR-01 を適用。新規コード追加時は必ず既存パターンをgrepで確認。
