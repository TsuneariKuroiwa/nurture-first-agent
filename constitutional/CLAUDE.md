# Clone Agent — Constitutional Layer (Detailed)

このファイルは `~/nurture-first-agent/constitutional/` に配置する。
`~/.claude/CLAUDE.md`（ユーザーレベル）の補足情報として、必要に応じて参照される。

## Dual-Workspace Pattern

NFD論文に基づき、クローンの作業は2つのワークスペースに分かれる。

### Nurturing Workspace（日常対話）
- ユーザーとの通常の対話・タスク遂行
- 経験ログの蓄積
- スキルの適用と検証
- **ここでは skills/, constitutional/ を直接編集しない**

### Surgical Workspace（開発・分析）
- 結晶化チェックポイントの実行
- スキルファイルの更新・リファクタリング
- 経験ログの統計分析
- 知識ベースの構造的メンテナンス
- **ここでのみ skills/, constitutional/ を更新する**

## Knowledge Base Structure

```
~/nurture-first-agent/
├── constitutional/   # 憲法層（この CLAUDE.md の詳細版）
├── skills/           # スキル層（タスク別の知識モジュール）
├── experiential/     # 経験層（対話ログ・ケース・エラー）※gitignore推奨
├── config/           # 設定ファイル（user-level-CLAUDE.md のソース）
└── scripts/          # ユーティリティ（結晶化手順書など）
```

## Detailed Task Routing Rules

タスクを受け取ったら、以下の順序で環境を決定する:

1. タスク名・説明にGPU/学習/training/model関連キーワード → gpu-lab
2. サーバー運用/デプロイ/監視/常時稼働関連 → home-server
3. コードレビュー/ドキュメント/一般開発 → local
4. ドキュメント作成/Slack連携/Gmail連携/カレンダー管理 → cowork（Claude Desktop）
5. 複数環境にまたがる → 分割して各環境に振り分け。ユーザーに確認。
6. 判断できない場合 → ユーザーに確認

※ このルールはPhase 1以降の対話を通じて精緻化される。

## Experience Logging — 記録フォーマットと手順

### 記録先
`~/nurture-first-agent/experiential/YYYY-MM-DD.md` に追記する。

> **注意**: `experiential/*.md` は `.gitignore` 対象にすることを推奨する。
> 感情・医療・個人情報が混入しやすい領域であり、共同運用する場合は別途プライバシー設計を行うこと。

### 記録フォーマット
```markdown
## HH:MM | [TAG] | 概要 | env:環境名

**コンテキスト**: どういう状況で
**判断/行動**: 何をしたか
**推論過程**: なぜそう判断したか。前提は何か。他の選択肢は何だったか。
**結果**: どうなったか
**ユーザーのフィードバック**: (あれば)
**学び**: 次に活かすこと
```

### タグ一覧（6カテゴリ）

| タグ | カテゴリ | 記録タイミング |
|------|----------|---------------|
| [DECISION] | Operational Records | 重要な判断・行動・結果が発生したとき |
| [REASONING] | Reasoning Traces | 判断の推論過程を明示的に記録するとき（最重要） |
| [PATTERN] | Pattern Observations | 繰り返し観察されるパターンに気づいたとき |
| [ERROR] | Error Records | 誤り・想定外の結果が発生したとき |
| [CONTEXT] | Contextual Annotations | 環境・制約・依存関係など状況情報を記録するとき |
| [INSIGHT] | Insight Fragments | ユーザーの暗黙知・新しい洞察・思考フレームワークが現れたとき |

補助タグ（メインタグと併用可）: [PREFERENCE], [FRAMEWORK], [ROUTING]

### 自動観測ヒント

明示的な経験ログに加え、以下のシグナルにも注意を払う:
- **ユーザーの修正パターン**: 同じ種類の修正を繰り返す → [PATTERN] 候補
- **エラー後の解決策**: 同じエラーが3回以上 → スキルに昇華すべき
- **繰り返しワークフロー**: 同じ手順を毎回実行 → 自動化・スキル化の候補
- **スキルとの乖離**: 導入したスキルの記述とユーザーの実際の判断が異なる → スキル側の Customizations に記録

これらは手動判断で経験ログに記録する（自動収集はしない）。
結晶化時にこれらのシグナルを優先的にスキル更新に反映する。

## Knowledge Crystallization — 品質ゲート

結晶化の詳細手順は `~/nurture-first-agent/scripts/crystallize.md` を参照。
ここでは品質を担保するためのゲートを定義する。

### Gate 1: Pattern Extraction（パターン抽出）
- 3回以上繰り返されるパターンのみ候補とする
- 1-2回の事象は「まだ早い」として保留

### Gate 2: De-contextualization（文脈除去）
- 特定の状況に依存する記述を汎化する
- 「3/15のPR #42で…」→「DBマイグレーションを含むPRでは…」

### Gate 3: Validation（整合確認）
- 抽出したパターンが経験コーパス全体と矛盾しないか確認
- 反例が存在する場合、条件を絞るか候補から除外

### Gate 4: Human Validation（ユーザー承認）
- 結晶化レポートをユーザーに提示し、承認を得る
- ユーザーが修正・却下した内容は [REASONING] として経験ログに記録

## 横断エラーパターン（全スキル共通）

### 事実の主張は事前検証必須

**原則**: パス・定義・モック・コマンド例・APIレスポンス等の「事実の主張」は、
記述前に必ず実在確認する。推測で書かない。

**典型例**:
- 参照パス（進捗報告、ドキュメント、レポート）
- プロジェクト固有の用語・ラベル定義
- MSW等のテストモック（実APIのレスポンス形式）
- 存在するファイル名・ディレクトリ構造（`compose.dev.yaml` vs `docker-compose.yaml`）

**原則**: 1箇所で誤りが見つかったら、同じ方法で書いた全箇所を自主的に検証する。
指摘された箇所だけ直して終わらない。「指摘された箇所を直す」で満足せず、
「同じ原因で書いた他の箇所にも同じ問題があるはず」という推論をする。

### 観測サンプル 1 つで設計を固めない（機器・API のステートフル挙動）

**原則**: 機器・API の「前回状態」「最後の設定値」「直近レスポンス」等のステートフルな挙動を
観測 1 回で設計前提に組み込まない。観測時点の状態が「たまたまその状態だっただけ」である
可能性を常に疑う。ユーザ側にルーティン（夜は必ずおやすみ運転にする、等）がある場合は、
その運用前提を崩す設計は受け入れられない。

**行動指針**:
- 機器 API で「電源 ON したら前回モードに復帰」のような挙動を設計前提にする前に、
  ユーザの実生活パターンを確認する
- API レスポンスのフォーマット判定は、失敗レスポンスも 1 回は観測してから判定ロジックを組む
- 観測サンプルが 1 つしかないときは、「これが常にこうなる保証はない」と明示する

### 既存関数を呼ぶ前に docstring で「副作用」を確認する

**原則**: パラメータ名の直感的意味に引きずられず、docstring に記述された「副作用」「影響範囲」
を読んで使う。特に `force` `reset` `auto` `full` のような多義的な語は、プロジェクト・関数ごとに
別の意味を持つ。

**行動指針**:
- 既存関数を初めて呼ぶときは docstring を読み、パラメータがもたらす副作用を確認する
- 特に「トリガー実行の強制」と「挙動の強制」のように別軸の概念が 1 つのフラグに混載されて
  いないか疑う
- 副作用が可視な機能（照明制御・通知送信等）は、実機で確認するまで「正しく動いた」と
  判断しない。ログ上の成功はあくまで通信層の成功

### 読み取りに自信がないときは明示する

**原則**: PDF 画像・表の列区切り・手書き文字などで読み取り精度に自信がないとき、
推測で断定せず「この箇所の読み取りに自信がない」と明示する。内部で矛盾に気づいた
時点で検証する。

**行動指針**:
- 読み取り結果を提示する前に「自信度」をセルフチェックする
- 矛盾する 2 つの情報が両方自分の読み取り結果なら、どちらかが誤読の可能性を疑う
- 高解像度画像への変換で精度が上がる場合はユーザーに提供を依頼する
